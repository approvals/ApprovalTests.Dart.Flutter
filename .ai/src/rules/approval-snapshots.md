# Approval Snapshot Rules

The package exists to produce snapshots that are stable and reviewable. Output that changes between runs defeats the purpose.

## Determinism

- Sort snapshot lines before emitting them (`widgetsString` sorts `metaLines`); output must not depend on widget-tree traversal order.
- Emit a full, self-contained snapshot on every `approvalTest()` call — pass `compareWithPrevious: false` so a second call in one test is not a delta of the first.
- Omit geometry (rects, transforms, scroll offsets) from semantics snapshots so output stays stable across screen sizes and platforms.
- Keep captured values free of timestamps, hash codes, memory addresses, and locale-dependent formatting.

## Approved vs received lifecycle

- Commit `*.approved.txt` (and approved `.png` goldens) next to their test files — they are the reviewed contract.
- Leave `*.received.*` files and the `.approval_tests/` cache (`class_names.txt`, `received_files.txt`) out of git; they are regenerated.
- Generated approved and cache files carry the header `# This file was generated by approval_tests. Please do not edit.` — let the tooling write them.

## Changing output

- Treat any change to snapshot text as a breaking change for consumers — re-approve affected files (`dart run approval_tests:review`) and note it in CHANGELOG.md.
- When you intend to change golden images, regenerate with `flutter test --update-goldens` and review the pixel diff before committing.
